# 使用 AI 从 0 到 1 生成完整项目：方法、流程与实战

## 1. 文档目标

本文是一份工程技术复盘，目标是讲清楚：一个完整的前后端项目，如何通过 VS Code 内的 AI 编程扩展，从无到有持续生成、验证、修正并最终交付。

重点不是“AI 会不会写代码”，而是“人如何指挥 AI 完成工程化交付”。

## 2. 先纠正一个常见认知：聊天式 AI 不等于工程式 AI

很多人对 AI 的使用停留在网页对话：你一句我一句、联网搜资料、写段文章。这种方式有价值，但做工程时有明显天花板：

1. 上下文窗口有限，跨文件、跨模块记忆能力弱。
2. 不能直接操作本地工程和运行环境。
3. 不会自动执行构建、调试、回归验证。
4. 输出通常是“建议文本”，不是“可运行结果”。

在 VS Code 中使用 CodeGeeX 等扩展后，AI 能力从“对话助手”升级为“工程代理”：

1. 可以读取并理解整个代码仓库上下文（小中型任务几乎不受限）。
2. 可以一次性改动多个文件，生成上千行代码。
3. 可以调用终端执行安装、构建、测试、健康检查。
4. 可以根据运行报错继续 Debug，不是只停留在口头建议。
5. 可以通过 API 接入网页端接触不到或难以直接使用的高级模型能力。

结论：从“聊天 AI”切到“工程 AI”，才是从无到有构建项目的关键分水岭。

## 3. 三种工作模式：Ask、Plan、Agent 怎么分工

常见扩展一般有三类模式（命名可能略有差异）：

1. Ask 模式
2. Plan 模式
3. Agent 模式

推荐分工如下：

1. 从 0 到 1 的大项目，用 Plan 模式起步。
2. 计划执行与联调阶段，用 Agent 模式持续落地。
3. 局部知识问答、原理解释、方案比选，用 Ask 模式。

### 3.1 为什么从 0 到 1 要优先 Plan 模式

Plan 模式的核心价值是“先做任务拆解和对齐，再写代码”，可以显著降低 vibe coding 常见风险：AI 理解偏题、自由发挥过度、结果不符合预期。

Plan 模式的典型流程：

1. 用户给出完整目标与约束。
2. AI 启动多个子代理拆解大任务（通常拆成 2-4 个子任务）。
3. AI 输出阶段计划、依赖关系、风险项和验收标准。
4. 用户确认或修改计划。
5. AI 按计划进入执行链路。

优点：

1. 先对齐“做什么”，再对齐“怎么做”。
2. 降低返工率，避免直接开写导致方向错误。
3. 能把复杂任务拆成可验证的小里程碑。
4. 适合连续执行几十分钟到数小时的大型构建。

### 3.2 Agent 模式在什么阶段更高效

当主工程搭起来后，Agent 模式特别适合：

1. 增加新接口或新页面。
2. 修复单点 bug。
3. 体验微调与样式修正。
4. 追加日志、导出、校验、防呆等增强项。

## 4. 从无到有前，先写“可执行需求”而不是“模糊愿望”

AI 最怕“描述很热闹，约束很缺失”。

在启动 Plan 前，建议把需求写成以下结构：

1. 业务目标：系统要解决什么问题。
2. 功能边界：必须做什么、明确不做什么。
3. 技术边界：前后端技术栈、数据库、部署方式。
4. AI 边界：接哪家模型、接口文档来源、调用约束。
5. 交互边界：设计规范、可视化风格、响应式要求。
6. 安全边界：鉴权、权限、隐私与脱敏要求。
7. 验收标准：必须可演示、可构建、可回归验证。

### 4.1 本项目在需求描述中的关键做法

项目起始阶段直接把四大模块写清楚并分点提交给 AI：

1. 家校沟通系统：消息推送、家长咨询自动回复、回执与协同。
2. 生涯规划与选课系统：基于成绩、兴趣、职业倾向给出建议。
3. 学业成长追踪系统：成长档案、趋势分析、风险预警。
4. 教学教研管理系统：任务、成果、绩效、流程管理。

同时明确“必须深度接入国产模型”，并提供 BigModel 官方 API 文档作为约束输入。

这个动作非常关键：

1. 可显著降低模型幻觉概率。
2. 避免 AI 写出伪造字段和错误调用方式。
3. 让接口实现可对照官方文档逐项验收。

## 5. 设计规范先行：不让 AI 自由发挥 UI

前端质量经常决定第一印象。

如果只说“做个好看页面”，AI 常会产出模板化界面：卡片套卡片、蓝紫渐变、风格同质化严重。

本项目直接提供了完整设计规范文档（DESIGN.md），按“先定视觉系统，再做页面”执行。关键约束包括：

1. 暖色纸张感主背景与暖灰中性色体系。
2. Serif 标题 + Sans 正文的排版层级。
3. Ring Shadow 与圆角尺度统一，而非重阴影堆叠。
4. 明确禁止冷色科技风和泛化渐变审美。
5. 规定响应式断点与移动端可用性。

这一步的本质是把“审美”变成“可执行约束”。

## 6. 架构决策：必须强调前后端分离

很多人让 AI 直接“做个系统”，AI 容易给出纯前端假实现（数据写死在浏览器），演示可看但不可运营。

本项目从第一轮就强调前后端分离，并明确职责：

1. 前端：页面交互、状态管理、可视化展示。
2. 后端：业务规则、权限控制、数据库读写、AI 网关代理。
3. 数据层：SQLite 建表、初始化、导入导出。

同时要求先设计 API 契约，再让 AI 按接口落地页面，避免“页面先跑了，接口对不上”。

## 7. 版本管理策略：每个可运行节点都提交

AI 连续改代码时，若上下文出现偏移，可能导致局部崩盘。没有版本管理会导致“能跑的版本回不去”。

推荐策略：

1. 完成一个可运行里程碑就提交一次。
2. 提交信息写清改动范围与风险点。
3. 遇到回归失败时，快速回退到上一个可行版本。
4. 参赛或业务敏感项目默认私有仓库。

这不是“管理洁癖”，而是 AI 工程化的最低保险丝。

## 8. 可复用的“指挥 AI 干活”操作手册

### 8.1 启动阶段（0 到 1）

1. 写总目标 + 模块清单 + 验收标准。
2. 写技术约束（栈、数据库、部署、模型来源）。
3. 附官方 API 文档与关键设计文档。
4. 使用 Plan 模式生成计划并逐项审查。
5. 确认后让 Agent 执行计划。

### 8.2 执行阶段（持续构建）

1. 每次只推进一个里程碑（例如先认证，再模块，再 AI）。
2. 每轮都要求 AI 给出“改动文件清单 + 验证命令 + 结果摘要”。
3. 失败时先让 AI 自查日志与报错，不直接人工接管。
4. 功能完成后，立即跑构建、接口健康检查和关键路径回归。

### 8.3 收尾阶段（可交付）

1. 补齐文档：README、使用手册、部署脚本、评比材料。
2. 补齐证据：审计日志、导出报表、操作脚本。
3. 补齐治理：许可证、AI 内容标识、隐私说明。

## 9. 本项目的实际生成路径（可直接复述）

可按以下时间线讲“从无到有”：

1. 初始化工程骨架：前后端分离目录、部署目录、文档目录。
2. 后端先行：认证、权限、四大模块 API、SQLite 数据层。
3. 前端接入：登录注册、角色导航、模块页面联动。
4. AI 集成：智谱模型网关、模型切换、模板调用。
5. 工程增强：导入导出、审计日志、班主任工作台、证据包。
6. 会话化升级：AI 历史会话、上下文续聊、模板脱敏。
7. 体验修复：结构化中文化、模型兼容过滤、思考链折叠展示。
8. 全链验证：构建通过、健康检查通过、真调用回归通过。

## 10. 如何减少幻觉与返工：六条硬规则

1. 规则先行：先给 AI 规则文件，再让 AI 写代码。
2. 文档约束：对外部 API 一律提供官方文档，不靠 AI 猜。
3. 分层执行：架构与契约先定，再做页面和细节。
4. 小步快跑：每轮改动后立刻验证，避免问题滚雪球。
5. 输出证据：让 AI 不仅说“已完成”，还要给运行证据。
6. 版本兜底：每个稳定节点提交，保证可回退。

## 11. 面向教育场景的额外注意事项

1. 学生数据最小化与脱敏输入。
2. AI 输出只做辅助，不替代教师专业判断。
3. 结构化输出优先，降低“长文本不可核验”风险。
4. 保留审计链路，确保关键操作可追踪。
5. 模型能力差异前置提示，避免用户误选导致失败。

## 12. 一个可直接复用的 Plan 模式输入模板

可按下面模板指挥 AI：

```markdown
你现在是我的全栈工程代理，请先进入 Plan 模式，不要立即写代码。

项目目标：

- [写业务目标]

必须实现的模块：

1. [模块A]
2. [模块B]
3. [模块C]

技术约束：

- 前端：[React/Vue + 版本]
- 后端：[Node/Python + 框架]
- 数据库：[SQLite/MySQL/PostgreSQL]
- 部署：[本地/服务器方式]

AI 约束：

- 模型供应商：[例如智谱]
- API 文档：[给出链接或附件]
- 禁止臆造字段，严格按文档实现

设计约束：

- 严格遵循 DESIGN.md
- 必须响应式，移动端可用

交付要求：

1. 先输出任务拆解、依赖关系、风险项、验收标准
2. 待我确认计划后再执行
3. 执行时每轮给出：改动文件、验证命令、验证结果
4. 完成后输出：启动方式、演示路径、风险与后续优化建议
```

## 13. 结语

“AI 生成项目”不是一句提示词换一份代码，而是一套可复制的工程管理流程：

1. 需求结构化。
2. 约束前置化。
3. 计划可审查。
4. 执行可验证。
5. 版本可回退。

把这五件事做好，AI 才能从“会聊天”变成“能交付”。
